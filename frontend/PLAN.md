# 2026년 6~8월 프론트엔드 스터디 계획

## 1. 스터디 목표

UMC 10기 Web 워크북과 미션에서 다뤘던 주제를 출발점으로 삼되, 그대로 반복하기보다 부족했던 기반 지식과 실제 프로젝트에서 설명해야 할 판단 기준을 보완한다.

8월 말에는 단순히 React 프로젝트를 만들어본 상태가 아니라 다음을 설명하고 증명할 수 있는 상태를 목표로 한다.

- 시맨틱 HTML과 CSS 레이아웃을 사용해 반응형 UI를 구현할 수 있다.
- 브라우저가 HTML, CSS, JavaScript를 받아 화면을 그리는 과정을 설명할 수 있다.
- HTTP 요청부터 브라우저 렌더링까지의 흐름을 설명할 수 있다.
- React의 렌더링, 상태, Effect, 컴포넌트 설계 원리를 설명할 수 있다.
- 클라이언트 상태와 서버 상태를 구분하고 적절한 관리 방법을 선택할 수 있다.
- TypeScript로 컴포넌트, API 응답, 상태의 타입을 안전하게 설계할 수 있다.
- Vite와 Webpack의 역할, 개발 서버와 프로덕션 빌드의 차이를 설명할 수 있다.
- 접근성, 성능, 테스트를 고려해 기능을 구현할 수 있다.
- CSR, SSR, SSG와 hydration의 차이를 설명할 수 있다.
- GitHub Actions를 이용해 lint, typecheck, test, build를 자동화할 수 있다.
- 기술 선택 이유와 트레이드오프를 면접에서 자신의 프로젝트 사례로 답할 수 있다.

## 2. 전체 학습 구조

프론트엔드 학습은 다음 다섯 축으로 진행한다.

```text
1. 기존 UMC 학습 경험
   └── 부족했던 개념과 구현 판단 기준 보완

2. 웹 기반 지식
   ├── HTML·CSS·접근성
   ├── 브라우저 렌더링
   ├── HTTP·네트워크·보안
   └── 웹 성능

3. React와 TypeScript
   ├── 렌더링과 상태 모델
   ├── Effect와 컴포넌트 설계
   ├── 타입 모델링
   └── 상태 관리

4. 프론트엔드 생태계
   ├── Vite·Webpack
   ├── 테스트
   ├── Git·CI/CD
   └── SSR·Next.js

5. 면접과 포트폴리오
   ├── fe-interview 문서
   ├── 기술 선택 이유
   ├── 트레이드오프
   └── 측정 가능한 개선 경험
```

UMC 워크북은 참고 자료이지, 7~8월 학습의 전체 범위나 과제 목록이 아니다.

## 4. 공부 방법

### 한 주제를 네 단계로 공부한다

```text
개념 확인
→ 작은 예제로 검증
→ 작은 실험 또는 프로젝트에 적용
→ 면접 답변으로 정리
```

예를 들어 `useEffect`를 공부할 때는 다음 순서를 따른다.

1. Effect가 필요한 경우와 필요 없는 경우를 공식 문서에서 확인한다.
2. 이벤트 구독과 정리 함수를 포함한 작은 예제를 작성한다.
3. 프로젝트의 API 요청 또는 이벤트 등록 코드를 점검한다.
4. "`useEffect`는 언제 사용해야 하나요?"에 2분 이내로 답한다.

### 매 회차 남길 결과물

각 회차마다 다음 네 가지를 남긴다.

1. 핵심 개념 노트
2. 직접 작성한 최소 예제
3. 프로젝트 적용 또는 기존 코드 분석
4. 면접 질문과 자신의 답변

### 자료의 우선순위

1. 공식 문서와 웹 표준
2. 기존 UMC 워크북과 미션 기록
3. 검증된 서적과 기술 문서
4. 블로그와 영상

기존 학습 자료와 `fe-interview` 문서에서 공식 문서와 다른 설명을 발견하면 공식 문서를 기준으로 수정한다.

### 학습 완료 기준

다음 세 조건을 만족해야 해당 주제를 공부한 것으로 본다.

- 코드를 작성할 수 있다.
- 동작 원리를 자신의 말로 설명할 수 있다.
- 언제 사용하고 언제 사용하지 않을지 판단할 수 있다.

단순히 문서를 읽거나 예제를 따라 친 것은 완료로 계산하지 않는다.

## 7월 1주차: HTML·CSS·접근성

### 목표

UMC Chapter 00을 그대로 다시 만들지 않고, 이미 만들었던 화면이나 간단한 카드/폼 화면을 더 정확하게 구현한다.

### 실험

- 같은 UI를 `div` 중심 구조와 시맨틱 구조로 각각 작성하고 차이를 비교한다.
- Flexbox와 Grid를 같은 레이아웃에 적용해보고 어떤 상황에 더 적합한지 기록한다.
- `position`, `z-index`, stacking context가 헷갈리는 예제를 직접 만든다.
- 폼에 `label`, 오류 메시지, focus style을 적용하고 키보드만으로 사용한다.
- Lighthouse Accessibility와 브라우저 Accessibility Tree를 확인한다.

### 반드시 답할 질문

1. 시맨틱 HTML은 단순히 태그 이름을 바꾸는 것 이상의 의미가 있는가?
2. Flexbox와 Grid는 어떤 기준으로 선택하는가?
3. `z-index`가 예상대로 동작하지 않는 대표 원인은 무엇인가?
4. 버튼처럼 보이는 요소와 실제 `<button>`은 어떤 차이가 있는가?
5. 접근 가능한 폼 오류 메시지는 어떻게 연결하는가?

### 결과물

- `semantic-html-and-accessibility.md` 초안
- 레이아웃 비교 예제 1개

## 7월 2주차: 브라우저·HTTP·개발자 도구

### 목표

브라우저가 요청을 보내고 화면을 그리는 과정을 말로만 외우지 않고, 개발자 도구에서 직접 확인한다.

### 실험

- 웹페이지 하나를 열고 document, JS, CSS, image 요청을 구분한다.
- 요청·응답 헤더, 상태 코드, 캐시 여부를 Network 탭에서 확인한다.
- DOMContentLoaded와 load 타이밍을 비교한다.
- 이벤트 캡처링, 버블링, 이벤트 위임 예제를 작성한다.
- `async`와 `defer`가 스크립트 실행 순서에 주는 영향을 작은 HTML로 검증한다.
- Performance 탭에서 style, layout, paint가 보이는 예제를 확인한다.

### 반드시 답할 질문

1. URL 입력 후 화면이 보이기까지 어떤 단계가 있는가?
2. DOM, CSSOM, render tree는 어떤 관계인가?
3. layout, paint, composite는 어떻게 다른가?
4. 이벤트 위임은 왜 동적인 목록에서 유용한가?
5. CORS 에러는 요청이 서버에 도달하지 않았다는 뜻인가?

## 7월 3주차: React 렌더링과 상태

### 목표

React를 "컴포넌트로 화면을 만드는 도구" 수준에서 넘어서, 렌더링과 상태 업데이트가 어떤 기준으로 일어나는지 이해한다.

### 실험

- 같은 값을 state, props, derived value로 각각 두고 어떤 문제가 생기는지 비교한다.
- 배열을 직접 변경하는 코드와 불변 업데이트 코드를 비교한다.
- 리스트의 `key`를 index로 둔 경우와 안정적인 id로 둔 경우를 비교한다.
- `setState` 연속 호출에서 일반 업데이트와 함수형 업데이트의 차이를 확인한다.
- 조건부 렌더링에서 state가 보존되거나 초기화되는 경우를 만든다.

### 반드시 답할 질문

1. React에서 렌더링과 실제 DOM 변경은 같은 의미인가?
2. state를 직접 변경하면 왜 문제가 되는가?
3. 함수형 업데이트는 언제 필요한가?
4. `key`는 단순히 warning을 없애기 위한 값인가?
5. state를 어디에 둘지 어떻게 결정하는가?

## 7월 4주차: Effect·Router·API 흐름

### 목표

`useEffect`를 API 요청용 도구로만 외우지 않고, 외부 시스템과 동기화할 때 필요한 도구로 이해한다. Router와 URL 상태는 프로젝트에서 검색, 필터, 상세 페이지를 설계할 때 바로 연결한다.

### 실험

- Effect가 필요 없는 계산을 Effect로 작성했을 때 생기는 문제를 확인한다.
- 타이머나 이벤트 리스너에 cleanup을 넣고 빼서 차이를 확인한다.
- 검색어, 필터, 페이지 번호를 URL search params에 저장한다.
- API 요청의 로딩, 에러, 빈 결과 상태를 분리한다.
- 오래된 응답 무시 또는 `AbortController`로 요청 취소를 실험한다.

### 반드시 답할 질문

1. `useEffect`는 언제 필요한가?
2. Effect 의존성 배열을 임의로 비워두면 어떤 문제가 생길 수 있는가?
3. stale closure는 무엇인가?
4. URL을 상태로 사용하면 어떤 장점이 있는가?
5. API 요청 상태를 UI 상태와 섞으면 어떤 문제가 생기는가?

## 8월 목표

8월에는 새 과제를 많이 만들기보다 진행 중인 프로젝트에 7월 학습을 연결한다. 프로젝트가 함께 진행되므로, 과제 중심이 아니라 "작게 실험하고, 필요한 부분만 적용하고, 이유를 기록하는 방식"으로 운영한다.

핵심은 다음 세 가지다.

1. 프로젝트 기능을 만들면서 상태, API, 접근성, 성능을 함께 점검한다.
2. 측정하거나 비교한 기록을 남긴다.
3. 면접과 포트폴리오에서 과장 없이 설명할 수 있는 경험을 만든다.

## 8월 주차별 계획

| 주차 | 주제 | 학습 방식 | 결과물 |
| ---- | ---- | --------- | ------ |
| 1주차 | 프로젝트 구조와 상태 분류 | 현재 프로젝트 기능을 상태 종류별로 나누고 관리 위치를 결정 | 상태 분류표, 컴포넌트 경계 노트 |
| 2주차 | API·서버 상태·에러 처리 | 조회, 생성, 수정, 삭제 흐름의 로딩/에러/캐시를 관찰 | API 흐름도, 에러 처리 기준 |
| 3주차 | 접근성·성능 점검 | Lighthouse, Performance, 키보드 탐색으로 문제를 찾고 작은 개선 적용 | 개선 전후 기록 |
| 4주차 | 테스트·배포·면접 정리 | 핵심 흐름 테스트와 배포 구조를 정리 | 테스트 목록, README/면접 답변 |

---

## 8월 1주차: 프로젝트 구조와 상태 분류

### 실험

- 프로젝트의 상태를 지역 UI 상태, URL 상태, 공유 클라이언트 상태, 서버 상태로 분류한다.
- 같은 상태를 props, Context, URL, 서버 캐시 중 어디에 둘지 비교한다.
- 컴포넌트가 너무 많은 책임을 갖는 지점을 찾는다.
- 폼, 모달, 필터, 목록 같은 반복 패턴의 경계를 정리한다.

### 결과물

- 상태 분류표
- 컴포넌트 경계 노트
- 선택하지 않은 대안과 이유

---

## 8월 2주차: API·서버 상태·에러 처리

### 실험

- API 함수와 UI 컴포넌트의 책임을 분리한다.
- 로딩, 에러, 빈 결과, 성공 상태를 각각 UI로 표현한다.
- 같은 데이터를 여러 화면에서 사용할 때 중복 요청과 캐시가 어떻게 동작하는지 확인한다.
- 생성·수정·삭제 후 목록을 어떻게 갱신할지 비교한다.
- 실패한 요청을 사용자에게 어떤 메시지로 보여줄지 기준을 만든다.

### 결과물

- API 흐름도
- 에러 처리 기준
- 서버 상태와 UI 상태 분리 예제

---

## 8월 3주차: 접근성·성능 점검

### 실험

- 키보드만으로 주요 흐름을 진행한다.
- Lighthouse Accessibility와 Performance를 측정한다.
- Chrome Performance 탭에서 긴 작업이나 불필요한 렌더링을 찾는다.
- 이미지 크기, 폰트 로딩, route-level code splitting 후보를 확인한다.
- 실제 병목이 확인된 부분에만 memoization을 적용하거나 제거한다.

### 보고서 형식

```text
문제:
측정 도구:
개선 전 상태:
원인 가설:
적용한 변경:
개선 후 상태:
트레이드오프:
```

### 결과물

- 접근성 문제 2개 이상 점검 또는 수정
- 성능 문제 1개 이상 측정
- 개선 전후 기록

---

## 8월 4주차: 테스트·배포·면접 정리

### 실험

- 핵심 사용자 흐름을 테스트 대상으로 고른다.
- 단순 구현 세부사항보다 사용자가 보는 결과를 검증한다.
- 배포 환경과 로컬 환경의 차이를 정리한다.
- README에 기술 선택 이유와 실행 방법을 작성한다.
- 프로젝트에서 실제로 한 것, 작은 예제로만 검증한 것, 개념만 학습한 것을 구분한다.

### 결과물

- 핵심 흐름 테스트 목록
- README 또는 프로젝트 설명 문서
- 기술 선택 이유
- 면접 답변 초안

# 8. 주간 운영 방식

## 권장 주간 루틴

프로젝트 일정과 함께 운영할 수 있도록 프론트엔드 공부는 주 4회로 제한한다.

| 요일 유형          | 수행 내용                    | 권장 시간 |
| ------------------ | ---------------------------- | --------: |
| 개념 학습 1회      | 공식 문서와 원리 학습        |   60~90분 |
| 실험·실습 1회      | 작은 예제와 동작 비교        |  90~120분 |
| 프로젝트 적용 1회  | 실제 기능 구현 또는 리팩터링 |   2~3시간 |
| 복습·면접 정리 1회 | 설명, 테스트, 문서화         |   60~90분 |

### 한 회차 운영

```text
10분: 오늘 답할 질문 작성
40분: 공식 문서와 기존 학습 자료 확인
60분: 최소 예제 또는 프로젝트 적용
20분: 면접 답변 작성
10분: 다음에 확인할 문제 기록
```

### 주간 계획 원칙

- 한 주에 핵심 주제는 최대 2개만 잡는다.
- 라이브러리 사용법보다 문제와 선택 이유를 먼저 적는다.
- 프로젝트에 적용하지 못한 개념은 작은 예제로 검증한다.
- 일요일에 다음 주 P1 작업 3개만 선정한다.
- 밀린 학습은 다음 주에 모두 이월하지 않고 중요도를 다시 평가한다.

# 9. `fe-interview` 저장소 운영 방식

`fe-interview`는 복사한 지식 저장소가 아니라, 검증한 답변을 축적하는 면접용 지식 베이스로 사용한다.

## 문서 작성 기준

각 문서는 다음 구조를 따른다.

```markdown
# 질문

## 한 문장 답변

## 동작 원리

## 예제

## 장점과 한계

## 언제 사용하는가

## 프로젝트에서의 경험

## 꼬리 질문

## 참고한 공식 자료
```

## 우선 작성할 문서

### 6월

- JavaScript 동적 타입과 타입 변환
- 원시 값과 객체의 전달
- 함수, 스코프, 호이스팅과 TDZ
- 실행 컨텍스트와 스코프 체인
- 클로저와 `this`
- 이벤트 루프와 Promise

### 7월

- 시맨틱 HTML과 접근성
- 브라우저 렌더링
- URL 입력부터 화면까지
- DOM 이벤트와 이벤트 위임
- React 렌더링과 state
- Effect와 외부 시스템 동기화
- URL 상태와 SPA routing

### 8월

- Core Web Vitals
- React 성능 최적화
- 접근성 점검
- API 에러 처리
- 서버 상태와 UI 상태 분리
- 프론트엔드 테스트
- SSR과 hydration
- 프로젝트 기술 선택
- 프로젝트 적용 기록

## 면접 답변 기준

답변은 다음 순서로 구성한다.

```text
정의
→ 동작 원리
→ 장단점
→ 선택 기준
→ 프로젝트 경험
```

경험이 없는 내용은 경험한 것처럼 작성하지 않는다. 대신 다음처럼 구분한다.

- 직접 구현한 경험
- 프로젝트 코드를 분석한 경험
- 작은 예제로 검증한 경험
- 개념만 학습한 상태

# 10. 최종 프로젝트에서 만들어야 할 경험

채용 공고의 요구를 단순 키워드가 아니라 실제 결과물로 연결한다.

| 채용 키워드  | 만들어야 할 증거                            |
| ------------ | ------------------------------------------- |
| HTML·CSS·JS  | 시맨틱·반응형 UI와 접근성 점검              |
| React        | 상태와 컴포넌트 설계가 포함된 프로젝트      |
| TypeScript   | strict mode와 안전한 API·상태 타입          |
| 상태 관리    | 상태 분류와 도구 선택 근거                  |
| Vite·Webpack | 빌드 흐름과 번들 분석 문서                  |
| 성능 최적화  | 개선 전후 측정 수치                         |
| 웹 표준      | 키보드·스크린 리더·시맨틱 점검              |
| SSR          | CSR·SSR 비교 예제와 hydration 설명          |
| WebView      | 연동 고려사항과 작은 검증 예제              |
| 개선 경험    | 기존 프로젝트를 단계적으로 보완한 기록      |
| Git·CI       | 작은 PR, 코드 리뷰, 자동화된 CI             |
| 협업         | 요구사항 변경과 의사결정 기록               |
| 문제 해결    | 문제 발견 → 원인 분석 → 해결 → 검증 문서    |

# 11. 학습 범위 조정 기준

## 시간이 부족할 때 반드시 유지

1. React 렌더링과 상태
2. HTTP·브라우저
3. TypeScript
4. 서버 상태와 API
5. 테스트
6. 접근성
7. 측정 기반 성능 개선

## 시간이 부족할 때 축소 가능

- Redux와 Zustand를 모두 깊게 구현하기
- Webpack 설정을 처음부터 직접 작성하기
- 고급 타입 체조
- 복잡한 애니메이션
- WebView 실제 앱 배포
- Next.js 전체 프로젝트 재작성

Redux Toolkit과 Zustand는 둘 다 사용법을 암기하기보다 하나를 실제 프로젝트에 적용하고, 다른 하나와의 선택 기준을 설명할 수 있으면 된다.

## 하지 않을 것

- 학습용 프로젝트를 여러 개 동시에 시작하기
- 측정 없이 `useMemo`와 `useCallback`을 추가하기
- 모든 상태를 전역 상태로 만들기
- 모든 API 요청을 Effect에 직접 작성하기
- 블로그 내용을 검증 없이 면접 답변으로 사용하기
- 라이브러리 사용 경험을 원리 이해로 착각하기
