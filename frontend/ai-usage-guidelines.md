# 개발 공부를 위한 AI 사용 가이드

## 1. 목적

AI를 사용하지 않는 것이 목표가 아니다. AI를 활용하면서도 다음 능력을 직접 기르는 것이 목표다.

- 문제를 스스로 정의하는 능력
- 공식 문서를 읽고 핵심을 찾는 능력
- 코드의 동작을 예측하는 능력
- 오류 원인을 추론하는 능력
- 기술 선택과 트레이드오프를 설명하는 능력
- AI가 생성한 답변의 오류를 발견하는 능력

AI 사용량보다 **AI를 사용하는 단계와 목적**을 제한한다.

> 먼저 혼자 생각하고, AI로 검증한 뒤, 다시 혼자 설명한다.

---

## 2. 기본 원칙

### 원칙 1. 첫 번째 답변은 직접 만든다

개념 질문, 설계, 면접 답변, 오류 원인 모두 먼저 자신의 답을 작성한다.

완벽할 필요는 없다. 다음과 같이 현재 생각을 짧게 적으면 된다.

```text
내가 알고 있는 내용:
확실하지 않은 부분:
예상하는 동작:
그렇게 생각한 이유:
```

AI에게 처음부터 완성된 설명이나 코드를 요청하지 않는다.

### 원칙 2. AI보다 공식 자료를 기준으로 판단한다

자료의 우선순위는 다음과 같다.

1. 공식 문서와 표준
2. 교재
3. 프로젝트 코드와 실행 결과
4. AI 설명
5. 블로그와 영상

AI와 공식 문서의 설명이 충돌하면 공식 문서와 실제 실행 결과를 기준으로 다시 확인한다.

### 원칙 3. 생성보다 검토에 사용한다

AI는 다음 역할에 우선 사용한다.

- 작성한 설명의 오류 검토
- 원인 가설의 타당성 확인
- 놓친 예외 상황 탐색
- 코드 리뷰
- 대안과 트레이드오프 비교
- 테스트 케이스 확장
- 꼬리 질문 생성

### 원칙 4. 설명할 수 없는 코드는 완료가 아니다

AI가 생성하거나 수정한 코드에 대해 다음을 설명할 수 있어야 한다.

- 이 코드가 해결하는 문제
- 입력과 출력
- 상태와 데이터의 흐름
- 이 컴포넌트나 함수가 분리된 이유
- 선택한 기술과 다른 대안
- 실패할 수 있는 상황
- 테스트 방법

설명하지 못하면 해당 코드는 자신의 학습 결과물로 계산하지 않는다.

### 원칙 5. AI 대화를 학습 결과물로 착각하지 않는다

AI와 긴 대화를 했다는 사실은 공부를 완료했다는 뜻이 아니다.

완료 여부는 다음 결과물로 판단한다.

- 직접 작성한 코드
- 자신의 말로 작성한 개념 노트
- 실행 결과 예측과 해설
- 테스트
- 프로젝트 적용
- 자료 없이 말한 면접 답변

## 3. 학습 단계별 사용 기준

| 학습 단계      | AI 사용 수준 | 권장 사용법                                             |
| -------------- | ------------ | ------------------------------------------------------- |
| 개념 첫 학습   | 낮음         | 공식 문서와 교재를 먼저 읽는다.                         |
| 개념 검증      | 중간         | 작성한 설명의 오류와 누락을 검토받는다.                 |
| 최소 예제 작성 | 낮음         | 먼저 직접 작성하고 실행 결과를 예측한다.                |
| 기능 설계      | 낮음         | 요구사항, 상태, 컴포넌트 구조를 먼저 직접 작성한다.     |
| 구현           | 중간         | 반복 코드와 API 사용법을 보조받는다.                    |
| 디버깅         | 중간         | 직접 원인 가설을 만든 뒤 검토받는다.                    |
| 코드 리뷰      | 높음         | 예외 상황, 복잡도, 접근성, 성능 문제를 탐색한다.        |
| 테스트         | 중간         | 기본 테스트는 직접 작성하고 누락 사례를 확장한다.       |
| 문서 표현 개선 | 높음         | 자신이 작성한 내용을 명확하게 다듬는다.                 |
| 면접 준비      | 제한적       | 질문과 꼬리 질문 생성에 사용하고 답변은 먼저 직접 한다. |
| 반복 작업      | 높음         | 보일러플레이트와 기계적 변경을 자동화한다.              |

## 4. 개념 공부에서의 AI 사용

### 권장 흐름

```text
공식 자료 읽기
→ 핵심 개념을 자신의 말로 작성
→ 예제의 실행 결과 예측
→ AI에게 오류와 누락 검토 요청
→ 공식 자료와 실행 결과로 재확인
→ 최종 노트 작성
→ 자료 없이 말로 설명
```

### AI를 사용해도 좋은 작업

- 어려운 문장을 쉬운 표현으로 다시 설명받기
- 유사한 두 개념 비교하기
- 자신의 설명에서 잘못된 부분 찾기
- 반례와 실행 결과 예측 문제 만들기
- 예상 면접 질문과 꼬리 질문 만들기

### 제한할 작업

- 공식 문서를 읽지 않고 AI 요약만 보기
- AI가 만든 설명을 그대로 노트에 복사하기
- 이해하지 못한 용어를 유지한 채 문서만 완성하기
- 출처가 없는 답을 면접 정답으로 암기하기

### 개념 검토 요청 템플릿

```text
학습 주제:

내가 이해한 내용:

직접 작성한 예제:

확실하지 않은 부분:

내 설명에서 사실과 다른 부분, 빠진 전제,
지나치게 단순화한 부분을 구분해서 알려줘.
완성 답변을 새로 작성하기보다 내 설명을 교정해줘.
```

## 5. 구현에서의 AI 사용

### AI를 사용하기 전에 직접 할 일

1. 요구사항을 한 문장으로 정의한다.
2. 입력과 출력을 적는다.
3. 필요한 상태를 분류한다.
4. 컴포넌트나 함수의 책임을 나눈다.
5. 구현 순서를 작성한다.
6. 첫 번째 코드를 직접 작성한다.

기능 구현의 첫 30~60분은 가능한 한 AI 없이 진행한다.

### AI를 사용해도 좋은 작업

- 반복되는 타입과 보일러플레이트 생성
- 익숙하지 않은 API의 사용 예제 확인
- 작성한 컴포넌트 구조 리뷰
- 예외 상황과 경계값 탐색
- 접근성 문제 점검
- 테스트 케이스 확장
- 리팩터링 대안 비교

### 제한할 작업

- 요구사항 전체를 붙이고 완성 기능 생성 요청
- 이해하지 못한 라이브러리 코드를 그대로 적용
- 설계 이유 없이 AI가 제안한 폴더 구조 사용
- 오류가 사라질 때까지 코드를 반복해서 붙여 넣기
- AI가 추가한 의존성을 검토 없이 설치하기

### 구현 요청 템플릿

```text
구현하려는 기능:

내가 정한 구조:

현재 코드:

내가 선택한 이유:

확실하지 않은 부분:

완성 코드를 바로 작성하지 말고,
현재 설계의 문제점과 구현 순서를 먼저 검토해줘.
필요하다면 최소한의 코드 조각만 제시해줘.
```

## 6. 디버깅에서의 AI 사용

### 20분 규칙

오류가 발생하자마자 AI에게 묻지 않는다. 최소 20분 동안 다음을 수행한다.

1. 오류 메시지를 처음부터 끝까지 읽는다.
2. 기대 결과와 실제 결과를 적는다.
3. 문제가 발생한 입력을 확인한다.
4. 브라우저 개발자 도구와 로그를 확인한다.
5. 최근 변경 사항을 확인한다.
6. 최소 재현 코드를 만든다.
7. 원인 가설을 하나 이상 작성한다.
8. 공식 문서에서 관련 API를 확인한다.

20분 후에도 해결되지 않으면 AI에게 현재 증거와 가설을 함께 전달한다.

### 디버깅 요청 템플릿

```text
현재 증상:

기대한 결과:

오류 메시지:

최소 재현 코드:

직접 확인한 내용:

내가 세운 원인 가설:

시도한 해결 방법:

정답 코드를 바로 주지 말고,
내 가설이 타당한지와 다음 확인 순서를 알려줘.
```

### AI 답변을 받은 후

1. 제안된 원인을 한 번에 하나씩 검증한다.
2. 코드 수정 전 실패를 재현한다.
3. 수정 후 같은 입력으로 다시 확인한다.
4. 왜 해결되었는지 한 문장으로 기록한다.
5. 가능하면 회귀 테스트를 추가한다.

## 7. 코드 리뷰와 리팩터링에서의 AI 사용

이 단계에서는 AI를 적극적으로 사용해도 된다. 단, AI의 의견을 자동으로 적용하지 않는다.

### 검토 항목

- 요구사항 충족 여부
- 컴포넌트와 함수의 책임
- 상태의 소유 위치
- 불필요한 Effect
- 타입 안정성
- 오류 처리
- 접근성
- 테스트 가능성
- 성능 병목
- 보안 문제
- 과도한 추상화

### 코드 리뷰 요청 템플릿

```text
이 코드의 목적:

중요한 제약 조건:

내가 고민한 부분:

다음 기준으로 리뷰해줘.

1. 실제 버그 가능성
2. 상태와 데이터 흐름
3. 타입 안정성
4. 접근성
5. 테스트 누락
6. 측정 없이 추가된 최적화
7. 과도한 추상화

스타일 취향보다 동작과 유지보수 문제를 우선해줘.
```

## 8. 테스트에서의 AI 사용

### 직접 작성할 부분

- 가장 중요한 정상 사용자 흐름
- 과거에 발생한 버그의 회귀 테스트
- 요구사항을 증명하는 핵심 테스트

### AI에 맡길 수 있는 부분

- 경계값 후보
- 실패 케이스
- 누락된 사용자 행동
- 반복적인 테스트 데이터
- 테스트 이름과 구조 개선

AI가 만든 테스트가 구현 세부사항만 확인하고 있지 않은지 검토한다.

### 테스트 요청 템플릿

```text
기능 요구사항:

현재 테스트:

테스트하지 못한 부분:

구현 세부사항이 아니라 사용자 행동과 결과를 기준으로,
누락된 정상·실패·경계 테스트를 분류해줘.
```

## 9. 면접 공부에서의 AI 사용

### 권장 흐름

```text
면접 질문 확인
→ 자료 없이 자신의 답변 작성
→ AI에게 사실 검증 요청
→ 공식 자료 확인
→ 답변 다시 작성
→ AI에게 꼬리 질문 요청
→ 자료 없이 소리 내어 답변
```

### AI를 사용해도 좋은 작업

- 질문 생성
- 답변의 사실 오류 검토
- 빠진 전제와 트레이드오프 탐색
- 꼬리 질문 생성
- 모의 면접 진행
- 답변을 더 간결하게 다듬기

### 제한할 작업

- AI가 만든 모범 답안을 먼저 읽고 암기하기
- 경험하지 않은 내용을 프로젝트 경험처럼 작성하기
- 이해하지 못한 전문 용어를 답변에 추가하기
- 말로 설명할 수 없는 긴 답변을 저장하기

### `fe-interview` 기록 기준

문서는 다음을 구분해서 작성한다.

- 직접 구현한 경험
- 기존 프로젝트에서 분석한 경험
- 작은 예제로 검증한 경험
- 공식 문서로만 공부한 내용

AI가 작성한 문장을 그대로 저장하지 않고, 자신이 실제 면접에서 말할 수 있는 길이와 표현으로 다시 작성한다.

### 모의 면접 요청 템플릿

```text
주제:

내 답변:

신입 프론트엔드 면접관처럼 진행해줘.

- 먼저 내 답변의 사실 오류만 짚어줘.
- 그다음 꼬리 질문을 한 번에 하나씩 해줘.
- 내가 답하기 전에는 모범 답변을 보여주지 마.
- 프로젝트 경험과 연결되는 질문도 포함해줘.
```

## 10. 성능 최적화에서의 AI 사용

AI에게 코드를 보여주고 바로 최적화해달라고 요청하지 않는다.

### 올바른 순서

```text
문제 관찰
→ 측정
→ 병목 가설
→ AI와 가설 검토
→ 최소 변경
→ 다시 측정
→ 결과와 트레이드오프 기록
```

### 직접 수행할 작업

- Lighthouse 결과 확인
- Performance trace 기록
- Network waterfall 확인
- React Profiler 측정
- 번들 크기 확인
- 개선 전후 수치 비교

### AI를 활용할 작업

- 측정 결과 해석 보조
- 가능한 원인 분류
- 실험 순서 설계
- 개선 방법의 부작용 검토
- 성능 보고서 표현 정리

측정 없이 `memo`, `useMemo`, `useCallback`을 추가하지 않는다.

## 11. 월별 AI 사용 기준

### 6월: 기반 원리 학습

AI 사용 강도: **낮음**

- 공식 문서와 교재를 먼저 읽는다.
- 실행 결과와 React 렌더링을 직접 예측한다.
- HTML과 CSS를 직접 작성한다.
- AI는 설명 교정과 문제 생성에 사용한다.

AI에게 맡기지 않을 것:

- JS·브라우저·React 핵심 개념의 첫 설명
- 미니 앱 전체 구현
- 면접 노트 초안 전체

### 7월: 서비스 기능 구현

AI 사용 강도: **중간**

- 요구사항과 상태 설계는 직접 한다.
- 인증과 API 흐름을 먼저 직접 그린다.
- 구현·디버깅·테스트에서 AI를 활용한다.
- 반복 코드와 테스트 데이터 생성은 적극적으로 맡긴다.

AI에게 맡기지 않을 것:

- 인증 저장 전략 결정
- 상태 관리 도구 선택
- API 오류 처리 정책
- 기술 선택 이유 작성

### 8월: 프로젝트 완성과 최적화

AI 사용 강도: **중간~높음**

- 반복 구현과 문서 정리에는 적극 사용한다.
- 성능 측정과 병목 판단은 직접 수행한다.
- AI에게 코드 리뷰와 반례 탐색을 요청한다.
- 포트폴리오 문장은 실제 구현 증거를 제공한 후 다듬는다.

AI에게 맡기지 않을 것:

- 성능 개선 전후 수치 생성
- 접근성 검증을 실제 테스트 없이 판단
- 프로젝트 의사결정과 트레이드오프 조작
- 경험하지 않은 성과 작성

## 13. AI 과사용 신호

다음 중 두 가지 이상에 해당하면 AI 사용 방식을 줄이거나 변경한다.

- 오류 메시지를 읽기 전에 AI에게 질문한다.
- AI 대화는 길지만 직접 작성한 코드와 노트가 적다.
- 같은 기능을 빈 파일에서 다시 만들 수 없다.
- 코드가 동작하지만 데이터 흐름을 설명하지 못한다.
- 설치된 패키지의 목적을 모른다.
- AI가 제안한 구조를 선택한 이유를 설명하지 못한다.
- 면접 답변에 자신이 평소 사용하지 않는 용어가 많다.
- 새로운 요구사항이 생기면 전체 코드를 다시 AI에게 요청한다.
- AI가 없는 상황에서 시작점을 잡지 못한다.
- 공식 문서보다 AI에게 먼저 질문하는 습관이 생긴다.

### 과사용 시 조치

1. 다음 학습 회차의 첫 60분은 AI를 사용하지 않는다.
2. 기존 코드를 보지 않고 핵심 예제를 다시 작성한다.
3. AI가 만든 코드를 한 줄씩 설명한다.
4. 불필요한 추상화와 의존성을 제거한다.
5. 자료 없이 3분 설명을 녹음한다.
6. 설명하지 못한 부분만 다시 공부한다.

- 글의 구조와 표현 개선
- 모의 면접과 꼬리 질문

### 조건부 허용

- 기능 코드 생성

- 타입 작성
- 테스트 생성
- 오류 수정
- 아키텍처 제안

조건부 허용 작업은 자신의 초안, 가설 또는 설계를 먼저 제공한 경우에만 사용한다.

### 학습 단계에서 금지

- 과제 전체를 한 번에 생성
- AI 답변을 검증 없이 제출
- 실행하지 않은 코드를 완료 처리
- 이해하지 못한 코드를 포트폴리오에 사용
- 경험하지 않은 성능 개선 수치 작성
- 경험하지 않은 협업과 기술 선택을 꾸며내기
- 공식 자료를 읽지 않고 면접 답변 암기

## 15. 최종 기준

AI 사용이 적절했는지는 사용 시간이나 질문 횟수로 판단하지 않는다.

다음 질문에 모두 답할 수 있으면 적절하게 사용한 것이다.

1. AI 없이 문제를 다시 정의할 수 있는가?
2. 코드와 동작 원리를 설명할 수 있는가?
3. 요구사항이 바뀌면 직접 수정할 수 있는가?
4. AI 답변의 오류를 의심하고 검증할 수 있는가?
5. 선택한 방법과 다른 대안의 트레이드오프를 설명할 수 있는가?
6. 실제 실행 결과와 테스트로 주장을 증명할 수 있는가?

> AI 없이 해당 코드와 기술 선택을 설명하거나 수정할 수 없다면, AI를 과도하게 사용한 것이다.
